Sorting and copying

We cover how to sort a DataFrame by one or more columns, how to reset the row index after sorting or filtering, and an important distinction between views and copies — and why it matters when modifying a DataFrame.

import pandas as pd

titanic = pd.read_csv("data/titanic.csv")

# ── Sorting with .sort_values() ──────────────────────────────

titanic.sort_values("fare", ascending=False)

# The original is unchanged
titanic.head(5)

# Assign back to store the sorted version
titanic = titanic.sort_values("fare", ascending=False)
titanic.head()

# ── Sorting by multiple columns ──────────────────────────────

titanic = titanic.sort_values(["pclass", "fare"], ascending=True)
titanic[["pclass", "fare", "survived"]].head(10)

# ── Resetting the index after sorting ────────────────────────

titanic.reset_index()

titanic = titanic.reset_index(drop=True)
titanic.head()

# ── The view vs copy distinction and copy() ──────────────────

# Filter out first-class passengers — may be a view in older pandas versions
first_class = titanic[titanic["pclass"] == 1]
first_class["fare_rounded"] = first_class["fare"].round(0)

# Best practice: use .copy() to create an independent DataFrame
first_class = titanic[titanic["pclass"] == 1].copy()
first_class["fare_rounded"] = first_class["fare"].round(0)

Exercise 3.5.4

This exercise is about understanding what .copy() does and why it matters depending on which version of pandas you are running.

Step 1. Check your pandas version by running:

import pandas as pd
print(pd.__version__)

Step 2. Read the following explanation, then answer the questions below.

When you filter a DataFrame, pandas may return either a view — a window into the original DataFrame’s memory — or a copy — an independent object. In pandas versions below 3.0, modifying a view could silently modify the original DataFrame, and pandas would raise a SettingWithCopyWarning to alert you. From pandas 3.0 onwards, filtering always returns a copy, so this warning no longer appears. However, calling .copy() explicitly is still good practice because it makes your intent clear to anyone reading your code.

Step 3. Run the code below and answer the questions.

import pandas as pd

titanic = pd.read_csv("data/titanic.csv")

# Without .copy()
first_class = titanic[titanic["pclass"] == 1]
first_class["fare_rounded"] = first_class["fare"].round(0)

# With .copy()
first_class_copy = titanic[titanic["pclass"] == 1].copy()
first_class_copy["fare_rounded"] = first_class_copy["fare"].round(0)
  1. Does your pandas version raise a SettingWithCopyWarning when running the first block? What does this tell you about your version?
  2. Check whether titanic has a fare_rounded column after running both blocks. What does this tell you about whether first_class and first_class_copy are independent of the original?
  3. Even if your version of pandas does not require .copy(), why may it still be a good habit to use it?
Solution 
import pandas as pd

titanic = pd.read_csv("data/titanic.csv")

# Without .copy()
first_class = titanic[titanic["pclass"] == 1]
first_class["fare_rounded"] = first_class["fare"].round(0)

# With .copy()
first_class_copy = titanic[titanic["pclass"] == 1].copy()
first_class_copy["fare_rounded"] = first_class_copy["fare"].round(0)

print("fare_rounded" in titanic.columns)
False

  1. If your pandas version is 3.0 or above, no warning appears. If you see a SettingWithCopyWarning, your version is below 3.0 and you should update pandas — run pip install --upgrade pandas in your terminal.

  2. titanic does not have a fare_rounded column in either case, meaning both first_class and first_class_copy are independent of the original on pandas 3.0+. On older versions, the first block without .copy() could have modified titanic directly.

  3. Even when not strictly required, .copy() can be good practice because it makes your intent explicit — anyone reading the code immediately knows this subset is meant to be independent. It also ensures your code behaves consistently if it is ever run on an older version of pandas.

This workflow — sort, reset index, filter with .copy(), modify — is one you will repeat constantly in data analysis. Each step serves a purpose: sorting makes the data easier to inspect, resetting the index avoids confusing label behavior with .loc[], and .copy() ensures modifications to the subset never silently affect the original DataFrame.
