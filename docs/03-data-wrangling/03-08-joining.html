Joining data

We cover the two main ways to combine separate tables into one: horizontal joining — bringing in new columns by matching rows on a shared key — and vertical stacking for combining tables that share the same columns.

import pandas as pd

healthexp = pd.read_csv("data/healthexp.csv")
healthexp.shape
healthexp.head()
healthexp["Country"].unique()
healthexp["Year"].nunique()

# Filter to a single year so each country appears only once
healthexp_2020 = healthexp[healthexp["Year"] == 2020].copy()
healthexp_2020

# ── Loading a CSV from a URL ─────────────────────────────────

# Repo: https://github.com/owid/owid-datasets/blob/master/datasets/Countries%20Continents/Countries%20Continents.csv
url = "https://raw.githubusercontent.com/owid/owid-datasets/refs/heads/master/datasets/Countries%20Continents/Countries%20Continents.csv"

continents = pd.read_csv(url)
continents.head()
continents["Entity"].nunique()

# ── Preparing the data for merging ───────────────────────────

continents = continents.drop(columns=["Year"])
continents = continents.rename(columns={"Entity": "Country"})
continents.head()

# ── Inner join ───────────────────────────────────────────────

merged_inner = healthexp_2020.merge(continents, on="Country", how="inner")
merged_inner

# ── Left join ────────────────────────────────────────────────

merged_left = healthexp_2020.merge(continents, on="Country", how="left")
merged_left

# ── Combining DataFrames vertically with concat() ────────────

titanic = pd.read_csv("data/titanic.csv")

# Split into two subsets
men   = titanic[titanic["sex"] == "male"].copy()
women = titanic[titanic["sex"] == "female"].copy()

print("Men:  ", len(men))
print("Women:", len(women))

# Stack them back into one DataFrame
combined = pd.concat([men, women])
print("Combined:", len(combined))
combined.head()

# Reset the index
combined = pd.concat([men, women]).reset_index(drop=True)
combined.head()

Exercise 3.8.1

The two DataFrames below contain exam scores for a group of students. Not all students took both exams.

import pandas as pd

exam1 = pd.DataFrame({
    "student_id": [1, 2, 3, 4, 5],
    "name":       ["Alice", "Bob", "Carol", "David", "Eva"],
    "score_q1":   [85, 92, 78, 90, 88]
})

exam2 = pd.DataFrame({
    "student_id": [1, 2, 3, 6, 7],
    "name":       ["Alice", "Bob", "Carol", "Frank", "Grace"],
    "score_q2":   [90, 88, 95, 72, 85]
})

Merge the two DataFrames using all four join types — inner, left, right, and outer — and answer the following questions for each:

  1. How many rows does the result have?
  2. Which students appear in the result, and which are excluded?
  3. Where do NaN values appear, and why?

Summarize in your own words when you would choose each join type.

Notice that merging on student_id produces duplicate name columns — name_x from the left table and name_y from the right. pandas adds suffixes automatically when both tables have a column with the same name that is not used as the merge key. You can avoid this by dropping the name column from one table before merging, or by merging on both student_id and name by passing a list of key columns to on.

  • Inner: 3 rows — only students 1, 2, 3 who appear in both tables. Students 4, 5, 6, 7 are silently dropped. Use when you only want observations present in both tables.
  • Left: 5 rows — all students from exam1. Students 6 and 7 are excluded; students 4 and 5 have NaN for score_q2. Use when your left table is the main dataset and you want to keep all of it.
  • Right: 5 rows — all students from exam2. Students 4 and 5 are excluded; students 6 and 7 have NaN for score_q1. Use when the right table is the one you want to preserve completely.
  • Outer: 7 rows — all students from both tables. NaN appears wherever a student is missing from one of the tables. Use when you want to keep everything and investigate what is missing on either side.

Exercise 3.8.2

In the previous exercise, merging on student_id alone produced duplicate name columns because both tables contained a name column that was not used as the key. One clean solution is to merge on multiple keys.

A colleague provides a second version of the exam data where the student names are stored differently:

import pandas as pd

exam1 = pd.DataFrame({
    "student_id": [1, 2, 3, 4, 5],
    "first_name": ["Alice", "Bob", "Carol", "David", "Eva"],
    "last_name":  ["Smith", "Jones", "White", "Brown", "Davis"],
    "score_q1":   [85, 92, 78, 90, 88]
})

exam2 = pd.DataFrame({
    "student_id": [1, 2, 3, 6, 7],
    "name":       ["alice smith", "bob jones", "carol white",
                   "frank miller", "grace wilson"],
    "score_q2":   [90, 88, 95, 72, 85]

The names in exam2 are in lowercase and stored as a single name column, while exam1 splits them across first_name and last_name.

  1. Add a name column to exam1 by combining first_name and last_name into a single lowercase string in the format "alice smith". Use the .str accessor.
  2. Merge the two DataFrames on both student_id and name using a left join. How does the result differ from merging on student_id alone?
  3. Drop the first_name and last_name columns from the result and print the final DataFrame.

Task 2: Merging on both keys avoids the duplicate name column problem from exercise 3.8.1. This is also safer: merging on multiple keys reduces the risk of false matches where two different people happen to share the same student ID or the same name.

Exercise 3.8.3

The code below merges two DataFrames that track product sales. Run it and inspect the result carefully.

import pandas as pd

products = pd.DataFrame({
    "product_id":   [1, 2, 3],
    "product_name": ["Coffee", "Tea", "Juice"]
})

sales = pd.DataFrame({
    "product_id": [1, 1, 2, 2, 3],
    "month":      ["Jan", "Feb", "Jan", "Feb", "Jan"],
    "units_sold": [120, 135, 80, 90, 60]
})

merged = products.merge(sales, on="product_id", how="left")
print("Products rows:", len(products))
print("Merged rows:  ", len(merged))
print(merged)
  1. How many rows does merged have compared to products? Why?
  2. Is this the result you would expect from a left join? What has happened?
  3. How would you check for duplicate keys in sales before performing the merge? Write the code to detect them.
  4. In this case, the extra rows are not a mistake — sales intentionally has multiple rows per product. How would you restructure the merge to get one row per product that includes both the product name and total units sold across all months?
Hint

For question 4, consider aggregating sales before merging rather than merging first and aggregating after.

Solution 
import pandas as pd

products = pd.DataFrame({
    "product_id":   [1, 2, 3],
    "product_name": ["Coffee", "Tea", "Juice"]
})

sales = pd.DataFrame({
    "product_id": [1, 1, 2, 2, 3],
    "month":      ["Jan", "Feb", "Jan", "Feb", "Jan"],
    "units_sold": [120, 135, 80, 90, 60]
})

merged = products.merge(sales, on="product_id", how="left")
print("Products rows:", len(products))
print("Merged rows:  ", len(merged))
print(merged)
Products rows: 3
Merged rows:   5
   product_id product_name month  units_sold
0           1       Coffee   Jan         120
1           1       Coffee   Feb         135
2           2          Tea   Jan          80
3           2          Tea   Feb          90
4           3        Juice   Jan          60
# 3. Check for duplicate keys in sales
print(sales["product_id"].value_counts())
product_id
1    2
2    2
3    1
Name: count, dtype: int64
# 4. Aggregate sales first, then merge
sales_total = sales.groupby("product_id")["units_sold"].sum().reset_index()
merged_clean = products.merge(sales_total, on="product_id", how="left")
print(merged_clean)
   product_id product_name  units_sold
0           1       Coffee         255
1           2          Tea         170
2           3        Juice          60

  1. merged has 5 rows — more than the 3 rows in products. Each product in products has been duplicated once for each matching row in sales.

  2. This is technically correct behaviour for a left join — every matching row in the right table generates a new row in the result. But it is often not what you want, and if you were not expecting it, the inflated row count is a sign that something needs attention.

  3. .value_counts() on the key column quickly shows whether any values appear more than once. In real data with thousands of rows, you can also check sales["product_id"].duplicated().sum() for a quick count of duplicate keys.

  4. Aggregating before merging is cleaner: first collapse sales to one row per product, then merge. This is a general principle — if the right table has multiple rows per key and you want a one-to-one merge, aggregate first.

Exercise 3.8.5

In this exercise you will merge two real datasets hosted on GitHub: one containing core passenger information from titanic and one containing passenger names and ticket details. Both datasets originate from the same source and share passenger_id as the primary key, but they also share two other columns — Pclass and Sex — with different casing.

import pandas as pd

names = pd.read_csv(url_names)

print(core.shape)

  1. Inspect both DataFrames. Which columns appear in both? How does the casing differ, and why does this matter for merging?
  2. Rename PassengerId, Pclass, and Sex in names to passenger_id, pclass, and sex so the column names match core.
  3. Merge the two DataFrames using a left join on passenger_id. Then drop the duplicate pclass and sex columns that result from the merge. Check the row count after merging to verify no rows were lost.
  4. Using the merged DataFrame, compute the survival rate by passenger class for male and female passengers separately. Which group had the highest survival rate?
Hint

After merging on passenger_id alone, pandas will suffix the duplicate pclass and sex columns as pclass_x, pclass_y, sex_x, and sex_y. You can drop the _y versions and rename the _x versions back, or alternatively merge on all three shared keys — passenger_id, pclass, and sex — to avoid the duplicates entirely.

Solution
